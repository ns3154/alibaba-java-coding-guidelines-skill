# 阿里巴巴 Java 开发规约摘要

来源：`https://alibaba.github.io/Alibaba-Java-Coding-Guidelines/`。本文件是便于 AI agent 使用的中文摘要，不替代官方原文。

## 级别含义

- 强制：默认必须遵守；审查时作为缺陷提出。
- 推荐：默认遵守；如果项目有充分理由偏离，说明原因。
- 参考：作为设计和治理建议，用于补充判断。

## 1. 编程规约

### 命名

- 不以下划线或美元符号开头或结尾。
- 命名不使用中文、拼音或拼音英语混合；中国专有名词可以用拼音。
- 类名使用 UpperCamelCase，通常是名词；DO、BO、DTO、VO 等领域模型后缀保持大写。
- 方法名、参数名、成员变量名、局部变量名使用 lowerCamelCase。
- 常量名使用大写字母加下划线，语义要完整。
- 抽象类以 `Abstract` 或 `Base` 开头；异常类以 `Exception` 结尾；测试类以被测类名开头并以 `Test` 结尾。
- 数组类型写成 `String[] args`，不要写成 `String args[]`。
- Boolean 字段不要以 `is` 开头，避免部分框架序列化属性名不一致。
- 包名全小写，点号后尽量是单数英文单词。
- 避免不常见缩写；设计模式可以体现在类名中，例如 `OrderFactory`、`LoginProxy`。
- Service 和 DAO 按 SOA 原则定义为接口，默认实现类以 `Impl` 结尾。
- 枚举类名以 `Enum` 结尾，枚举成员使用大写加下划线。
- 分层对象命名：DO 对应表，DTO 用于传输，VO 用于视图，BO 封装业务；不要用 `POJO` 作为类型后缀。

### 常量

- 禁止魔法值，预定义常量除外。
- long 或 Long 字面量使用大写 `L`，避免小写 `l` 被看成数字 1。
- 常量按功能归类到不同常量类，避免一个巨大的公共常量类。
- 固定范围值或带属性的值优先使用枚举。

### 格式

- 代码块有内容时，左大括号不换行，右大括号按语句结构换行；空块可以写成 `{}`。
- `if`、`for`、`while`、`switch` 等关键字和左括号之间有一个空格。
- 操作符左右各一个空格。
- 缩进使用 4 个空格，不使用 tab。
- 单行长度不超过 120 字符；换行时操作符、点号和逗号后的内容放到下一行并保持清晰缩进。
- 文本文件使用 UTF-8 和 Unix 换行。
- 不为了纵向对齐变量而插入多余空格；同一语义块之间用一个空行分隔。

### 面向对象

- 静态字段和静态方法通过类名访问。
- 覆写接口或抽象类方法必须加 `@Override`。
- 可变参数仅适合同类型同语义参数，且必须放在参数列表末尾。
- 对外接口废弃时加 `@Deprecated` 并说明替代方案；禁止继续使用废弃类或方法。
- 调用 `equals` 时优先使用常量或确定非空对象调用；包装类型比较使用 `equals`，不要使用 `==`。
- 浮点数等值判断不要直接用 `==` 或包装类 `equals`，使用误差范围或 `BigDecimal`。
- POJO 成员使用包装类型；RPC 返回值和参数使用包装类型；局部变量推荐使用基本类型。
- DO、DTO、VO 等 POJO 字段不要设置默认值。
- 序列化类兼容升级时不要随意改变 `serialVersionUID`。
- 构造方法中不要写业务逻辑，初始化逻辑放到 `init` 等方法。
- POJO 应实现 `toString`，便于排查问题。
- 使用 `split` 后按下标访问数组前，注意末尾分隔符导致的边界问题。

### 集合

- 重写 `equals` 时必须重写 `hashCode`。
- `subList` 返回的是原列表视图，原列表结构变化会影响子列表。
- `Arrays.asList` 返回的是固定大小列表，不能执行增加或删除。
- 不要在 foreach 循环中直接删除或新增集合元素；删除时使用迭代器。
- 集合初始化时尽量指定容量，减少扩容成本。
- Map 遍历优先使用 `entrySet`，需要 key 或 value 时再选择对应方式。
- 高并发场景注意集合线程安全，不要误用非线程安全集合。

### 并发

- 线程资源必须通过线程池提供，不显式创建大量线程。
- 线程池和线程要有有意义的名称，便于日志和问题定位。
- 线程池参数要按业务场景设置，不盲目使用默认工厂方法。
- 使用 `SimpleDateFormat` 等非线程安全对象时要隔离实例或改用线程安全方案。
- 高并发代码要明确锁粒度、共享状态、超时、拒绝策略和降级方案。

### 控制流

- `switch` 的每个 `case` 要通过 `break`、`return` 或注释说明有意贯穿。
- 表达异常分支时尽早返回，减少嵌套层级。
- 条件判断中复杂布尔表达式应拆出有语义的变量或方法。
- 避免过深嵌套；优先让主流程清晰可读。

### 注释

- 类、类属性、类方法上的注释使用 Javadoc 规范。
- 抽象方法、接口方法、复杂业务逻辑、特殊分支、算法和临界值要解释意图。
- 注释要和代码同步更新，避免误导。
- 代码注释使用中文；对外 API 可以按项目约定补充英文。

## 2. 异常和日志

### 异常

- 不捕获 `NullPointerException`、`IndexOutOfBoundsException` 等运行时异常来代替前置检查。
- 不把异常用于普通控制流。
- try-catch 范围要尽量小，稳定代码和不稳定代码分开。
- 不吞掉异常；不处理就继续抛出，由顶层转换为用户可理解的信息。
- 事务方法抛出异常时要保证回滚语义明确。
- 流、连接等资源在 finally 或 try-with-resources 中关闭；finally 中不要抛异常或 return。
- 捕获的异常类型要和抛出的异常相匹配。
- 方法可能返回 null 时必须在 Javadoc 或注释中说明，调用方要做空值判断。
- 不直接抛 `RuntimeException`、`Exception`、`Throwable`，优先定义清晰的业务异常。

### 日志

- 使用 SLF4J 门面，不直接使用 Log4j、Logback 等具体实现 API。
- TRACE、DEBUG、INFO 级别日志使用占位符或先判断日志级别，避免无谓字符串拼接。
- 异常日志要同时包含上下文信息和异常堆栈。
- 错误日志和业务日志尽量分离，便于排查和监控。
- 扩展日志文件命名体现应用名、日志类型和用途。
- 日志信息避免输出密码、密钥、完整手机号、身份证等敏感数据。

## 3. MySQL 规约

### 表结构

- 表达是否概念的字段使用 `is_xxx`，数据类型使用无符号 tinyint；POJO 属性不要带 `is` 前缀，需要映射。
- 表名、字段名使用小写字母和下划线，不以数字开头，不使用复数。
- 禁止使用 MySQL 保留字。
- 主键索引名为 `pk_字段名`，唯一索引名为 `uk_字段名`，普通索引名为 `idx_字段名`。
- 小数类型使用 `decimal`，不要用 `float` 或 `double` 存储精确数值。
- varchar 长度按实际需要分配，避免过度放大。
- 表必须有 `gmt_create` 和 `gmt_modified`，并在更新时维护修改时间。
- 单表行数过大或容量过大时，提前设计分库分表或归档策略。

### 索引

- 业务上唯一的字段必须建唯一索引。
- 三表以上 join 默认禁止；join 字段类型必须一致并建索引。
- varchar 建索引要指定合适长度。
- 分页查询禁止 `LIKE '%xxx'` 或 `LIKE '%xxx%'`，必要时使用搜索引擎。
- `ORDER BY` 尽量利用联合索引顺序。
- 优先使用覆盖索引减少回表。
- 大分页使用延迟关联或子查询优化。
- EXPLAIN 目标至少达到 range，最好达到 ref 或 const。
- 联合索引左侧优先放区分度高且参与等值条件的列。

### SQL

- 统计行数使用 `COUNT(*)`，不要用 `COUNT(列名)` 或 `COUNT(常量)` 替代。
- 注意 `COUNT(distinct ...)`、`SUM` 和 NULL 的语义。
- 判断 NULL 使用 `ISNULL()` 或 `IS NULL`，不要直接比较。
- 分页查询先查 count，count 为 0 时立即返回。
- 禁止外键和级联更新，关联逻辑由应用层处理。
- 禁止存储过程。
- 数据订正、删除和更新前先 SELECT 确认数据范围。
- 大量 IN 查询要评估集合大小，无法避免时控制在合理范围内。
- 国际化字符使用 UTF-8，表情符号需要考虑 UTF8MB4。
- 不推荐在业务代码中使用 TRUNCATE。

### ORM

- 查询必须写明具体字段，禁止直接 `SELECT *`。
- Boolean POJO 属性不以 `is` 开头，数据库字段可以以 `is_` 开头，通过 resultMap 映射。
- 不使用 `resultClass` 直接接收查询结果，应该定义明确 DO 并配置映射。
- MyBatis 参数使用 `#{}`，禁止用 `${}` 拼接用户输入。
- 不用 HashMap 或 Hashtable 作为数据库查询结果类型。
- 更新记录时同步更新 `gmt_modified`。
- 不定义“万能更新接口”，只更新明确需要更新的字段。
- 不滥用 `@Transactional`，注意事务范围、QPS、缓存回滚、消息补偿和统计调整。

## 4. 工程规约

### 分层

- 默认上层依赖下层：开放接口、终端展示、Web、Service、Manager、DAO、外部接口各司其职。
- Web 层负责访问控制、参数校验和不可复用的请求处理。
- Service 层承载业务逻辑。
- Manager 层封装第三方服务、通用能力、缓存、中间件处理和多个 DAO 的组合复用。
- DAO 层只处理数据访问，不承载业务决策。
- Open API 层使用错误码和错误信息处理异常。
- Query 对象承载查询条件；超过两个查询条件时禁止用 Map 代替。

### 依赖和版本

- GAV 命名要稳定：GroupId 不超过四级，ArtifactId 表示产品名和模块名。
- 版本号使用主版本、次版本、修订号；初始版本从 `1.0.0` 开始。
- 线上应用不依赖 SNAPSHOT 版本，安全包等特殊情况除外。
- 升级依赖前后对比依赖树，避免传递依赖冲突。
- 多个相关依赖使用统一版本变量；同一 GroupId 和 ArtifactId 在子项目中版本一致。
- 父 POM 用 `dependencyManagement` 管版本，子模块在 `dependencies` 中声明依赖。

### 服务器

- 高并发服务器关注 TIME_WAIT、文件描述符上限和连接资源。
- JVM 推荐配置 OOM dump，便于线上问题定位。
- 内部重定向使用 forward，外部重定向使用统一 URL 组装和安全校验。

## 5. 安全规约

- 用户专属页面和功能必须做权限校验，防止越权访问或操作。
- 用户敏感数据展示必须脱敏。
- 用户输入参与 SQL 时必须参数化或按元数据限制，禁止字符串拼接 SQL。
- 所有用户输入必须做参数校验，覆盖页大小、排序字段、跳转地址、反序列化、正则等风险。
- 用户数据输出到 HTML 页面前必须做安全过滤或转义。
- 表单和 AJAX 提交必须做 CSRF 校验。
- 短信、邮件、电话、订单、支付等资源使用场景必须做防重放、防刷和频控。
- 用户生成内容需要接入反欺诈、敏感词和风控策略。
